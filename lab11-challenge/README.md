# Lab 11 — Final Challenge solution

End-to-end GitOps scenario combining governance, packaging, delivery control, fleet scaling and troubleshooting. Maps to the tasks of `Challenge.md`.

## `argocd/`

### `project-challenge-platform.yaml`
`AppProject` **challenge-platform** — restricted non-production domain:

- `sourceRepos`: only `DSK213-PUBLIC`
- `destinations`: only namespaces matching `challenge-*` on the in-cluster Kubernetes
- `clusterResourceWhitelist`: `Namespace`
- `namespaceResourceWhitelist`: `Deployment`, `Service`, `ConfigMap`, `batch/Job`
- `orphanedResources.warn: true`

### `applicationset-webapp-fleet.yaml`
`ApplicationSet` **webapp-fleet** with a `list` generator producing one Application per environment:

- `webapp-dev` → namespace `challenge-dev`, values file `values-dev.yaml`
- `webapp-stage` → namespace `challenge-stage`, values file `values-stage.yaml`

Both Applications point to the same chart path (`charts/webapp`) and enable `automated` sync with `prune: true` + `selfHeal: true`. `CreateNamespace=true` ensures the target namespaces are provisioned.

## `charts/webapp/`

Helm chart **webapp** (`type: application`, `version: 0.1.0`, `appVersion: "1.27"`) — a minimal nginx workload that demonstrates both a values-driven packaging model and a PreSync hook with sync-waves.

### `values.yaml`
Defaults: nginx `1.27-alpine`, 1 replica, ClusterIP on port 80, `labels.env=default`, `config.greeting="hello from the default values"`, standard resource requests/limits.

### `values-dev.yaml`
Dev overrides: 1 replica, `labels.env=dev`, `config.greeting="hello from dev"`.

### `values-stage.yaml`
Stage overrides: 2 replicas, `labels.env=stage`, `config.greeting="hello from stage"`.

### `templates/_helpers.tpl`
Standard helpers: `webapp.name`, `webapp.labels` (including user-provided `labels` map), `webapp.selectorLabels`.

### `templates/configmap.yaml`
`ConfigMap` **webapp-config** with `argocd.argoproj.io/sync-wave: "-1"` — created before everything else. Holds the `greeting` value consumed by the Deployment via `envFrom`.

### `templates/pre-sync-job.yaml`
`Job` **webapp-presync** annotated with `argocd.argoproj.io/hook: PreSync`, `hook-delete-policy: BeforeHookCreation`, `sync-wave: "0"`. A busybox container logs a simulated schema migration (includes the current environment label) and exits. Runs **before** the workload is applied.

### `templates/deployment.yaml`
nginx Deployment annotated with `sync-wave: "1"`. Templated on replicas, image, resources and labels. Consumes `webapp-config` via `envFrom`.

### `templates/service.yaml`
ClusterIP Service annotated with `sync-wave: "1"`. Templated on type and port, targets the `http` named port on the pods.

## Break / Diagnose / Fix walkthrough

Important finding from the pilot: `webapp-dev` is generated by the `webapp-fleet` ApplicationSet. Any direct `kubectl patch` on a child Application is automatically reverted by the ApplicationSet controller, which reconciles the children against its template within seconds. To break a set-managed Application in a stable way, you have two options:

- **Option A (recommended)** — break from the template: edit `applicationset-webapp-fleet.yaml` so the failing element points to `values-dev-wrong.yaml`, commit, push, re-apply.
- **Option B** — delete the ApplicationSet, break the orphan Application, diagnose, fix, re-apply the ApplicationSet.

### Option A — break via template

1. In `applicationset-webapp-fleet.yaml`, change the `valuesFile` of the `dev` element:
   ```yaml
   - env: dev
     namespace: challenge-dev
     valuesFile: values-dev-wrong.yaml
   ```
2. Commit, push, re-apply:
   ```bash
   curl -sSL "<raw URL of the ApplicationSet>" -o /tmp/appset.yaml
   kubectl apply -f /tmp/appset.yaml
   ```
3. Follow the fast path:
   ```bash
   argocd app get webapp-dev --refresh
   argocd app diff webapp-dev
   kubectl logs -n argocd deploy/argocd-repo-server --tail=30 | grep -Ei "webapp|error|not found|failed"
   kubectl get deploy -n challenge-dev
   ```
   Expected:
   - `ComparisonError` from `repo-server` (`helm template ...values-dev-wrong.yaml: no such file or directory`)
   - Live Deployment still running in `challenge-dev` (not pruned)
4. Fix by reverting the element in Git, commit, push, re-apply.

### Observed pilot behaviour (direct patch)

Patching the child Application directly DOES trigger a transient failure in `repo-server` (the error is visible in the logs), but the ApplicationSet controller reverts `valueFiles` to the template value within seconds. The end user sees the Application as `Synced/Healthy` almost immediately. Evidence in the pilot:

- `kubectl patch application webapp-dev ... values-dev-wrong.yaml` succeeded.
- `argocd-repo-server` log line: `helm template ...values-dev-wrong.yaml failed: no such file or directory`.
- Within seconds, `argocd app get webapp-dev` showed `Helm Values: values-dev.yaml` again and `Sync Status: Synced`.
- A follow-up `kubectl patch ... values-dev.yaml` reported `patched (no change)` — the field was already back to the template value.

This is a safety property of `ApplicationSet`: children are immutable against out-of-band edits.

## Ownership reasoning

- **Source of truth**: Git (`DSK213-PUBLIC`). Only via commits do new desired states enter the system.
- **Promotion**: controlled via values files per environment or via image-updater policies. No one edits live workloads.
- **Reconciliation**: ArgoCD `application-controller`, bounded by `challenge-platform` project policy.
- **Safety for growth**: every new environment is an additional element in the `list` generator (or a labelled cluster secret for the `clusters` generator). Governance does not loosen — the project contract is enforced equally on every generated Application, as demonstrated in Lab 09.
